NLP & Language Models
The foundation everything is built on

Natural Language Processing (NLP)

A field of AI focused on developing technologies capable of understanding, interpreting, and generating human language.

Language Model (LM)

Not a specific model — it's a concept / architecture. A Language Model is any compute program that learns to predict the next word (or token) in a sequence, based on what came before.

💡 Real-world example: Email autocomplete — when Gmail suggests "I hope this email finds you well…" it's using a language model.
P(next_token | previous_tokens) ← Core Objective (Bayes' Theorem)
🎯 Goal: Find the probability of the next token given all previous tokens.

Tokenization & Token IDs
How text becomes numbers the model can process

Tokenization

The process of breaking down text into smaller units called tokens. Tokens can be words, subwords, characters, or even sentences — depending on the level of granularity required.

Example: "The white cat sat on the mat"

Step 1 — Tokenize into tokens

Tokenizer

A tool or algorithm that performs tokenization. Every language model has its own tokenizer specifically designed to work with that model.

Why model-specific tokenizers?

Different models use different vocabularies and splitting strategies. When using different models, you must use their respective tokenizers to ensure compatibility.

Each token ID maps to a fixed vector from the embedding table (learned during training).

Positional Encoding — Position of each token in the sequence

Transformers process all tokens in parallel — positional encoding tells the model where each token appears.

"cat" at position 3 gets a different encoding than "cat" at position 7.

This is the magic: each token's vector gets updated based on all other tokens. The vector for "cat" now reflects that it's a white cat sitting on something.

Static embedding of "cat" + context of all other tokens → rich contextual vector E3'

Self-Attention: Query, Key, Value
The core mechanism of transformers

Self-Attention

When other tokens "pay attention" to one specific token to understand its meaning in context, it's called self-attention.

For "The white cat sat on the ___": to predict the next word, each token must understand every other token. This is done through Q, K, V.
Q
Query

The token we want to find context for. If we want the contextual embedding for "cat", then "cat" is the Query.

Formula: W(Q) × E_cat = Q3 (query vector)
K
Key

All other tokens that are adding context to the query. In "The white cat sat on the", The, white, sat, on, the are Keys.

Formula: W(K) × E_white = K3 (key vector)
V
Value

The actual content contribution from each key token, weighted by relevance. Used to compute the final context update.

Formula: W(V) × E_cat = V3 (value vector)

Relevancy Score, Softmax & Masking
How attention weights are calculated and applied

Relevancy Score

Computed as the dot product of Q · K. Gives a raw score for how relevant each key is to the query.

score = Q · K
(cat_query · each_key_vector)

Softmax

Converts raw dot product scores (any real number) into probabilities that sum to 1. Closer to 1 = more relevant.

softmax(scores) → [0.1, 0.6, 0.2, 0.05, 0.05]
= attention weights (sum to 1)

Masking (Before Softmax) — The Key Decoder-Only Feature

In Decoder-Only transformers (GPT), we mask future tokens so "cat" can only attend to previous tokens, not future ones. This is what differentiates Decoder-Only from Encoder-Only.

Why? During training, the model must learn to predict the next token without "cheating" by looking ahead. At inference, future tokens simply don't exist yet.

Encoder-only (BERT) → bidirectional, can see all tokens at once ✅
Decoder-only (GPT) → unidirectional, sees only past tokens via masking 🔒

Delta Context (ΔE) — The Final Contextual Embedding

After masking + softmax, we multiply each Value vector by its attention weight and sum them up. This "delta" gets added to the static embedding:

ΔE3 = V1×0.1 + V2×0.6 + V3×0.2 + ...
= 10% The + 60% white + 20% cat + ...

E3' = E3 (static) + ΔE3 (context)
E3' is the final contextual embedding for "cat" — it now contains information about the white cat sitting on something.

Why the last token?

The last token has attended to ALL previous tokens through masked self-attention. It has the richest, fullest context of the entire sequence — making it the perfect representation for predicting what comes next.

Encoder-Only vs Decoder-Only vs Encoder-Decoder
Choosing the right architecture for the task

Encoder-Only (BERT)

  • Bidirectional context — sees all tokens simultaneously
  • Goal: Understand input deeply, no generation
  • No masking of future tokens
  • Best for: text classification, sentiment analysis, named entity recognition
  • Architecture: same as Decoder-Only but without output generation
BERT = Bi-Directional Encoder Representations from Transformers

Decoder-Only (GPT)

  • Unidirectional — only sees previous tokens (masked)
  • Goal: Generate text token by token
  • Masking ensures no future token leakage
  • Best for: text generation, chatbots, code generation
  • Is a Transformer → GPT Model → Decoder Only
GPT = Generative Pre-Trained Transformer

Encoder-Decoder (T5)

Uses both: Encoder creates richer embeddings of the input. Decoder then takes that + its own previous tokens to generate output.

Encoder (Step 1)
Richer Embeddings of entire input sentence using bidirectional attention
Decoder (Step 2)
Takes encoder output via Cross-Attention → Feed Forward → LM Head → generates output
Use case: Translation — "The white cat sat on the mat" → "Le chat blanc était assis sur le tapis" (T5 model)
